02 — Architecture

A live control plane, not a token vault.

Passport doesn't move tokens between providers — providers don't allow that. Passport routes execution to the provider that already holds the credential. Your AI app receives the result. The raw token never touches your code.

01 / Consent

User approves a Set, not a checklist.

A Set is a named bundle — Founder Stack, Engineering Stack, Client A. Users approve the Set for an AI product in one click. They see exactly what's requested before they say yes.

02 / Routing

The control plane picks the provider.

Each connection lives behind a provider — Composio, Arcade, Pipedream, Nango, or Zapier. Passport stores the mapping. When an action fires, Passport routes to the right one, swaps adapters as needed, and never exposes the underlying token.

03 / Audit

Every action is logged and revocable.

Users see what each AI product did, when, and against which account. One click revokes a product, a connection, or a whole Set. No support tickets. No phantom access.

03 — Developer SDK

Three lines of code between you and 9,000 apps.

Server SDK creates the session. The drop-in Connect UI handles consent. Your agent calls passport.execute() or hits the MCP gateway. We handle the rest.

Skip the auth plumbing.

One SDK across providers. Drop-in React component for consent. MCP server URL ready for Claude, ChatGPT, Cursor, or anything that speaks the protocol.

  • Provider-neutral. Swap Composio for Nango without rewriting your agent code.
  • White-label consent UI. Your users see your brand, not ours, throughout the OAuth flow.
  • Hosted MCP gateway. One URL, scoped to one user, with one Set of approved apps.
  • Webhooks for connection lifecycle: connection.active, connection.expired, passport.revoked.
  • Open-source spec. The Passport protocol is public — we run the hosted control plane.

08 — Open questions

Questions worth asking.

Are tokens portable between providers?
No, and we don't claim they are. Provider projects are isolated — a Gmail credential connected through Composio cannot be moved into Pipedream's vault. Passport doesn't move tokens. It stores the mapping of which provider holds which credential for which user, and routes execution to the right one. App-access portability, not token portability.
What stops Composio or Zapier from copying this?
Provider-neutrality. The moment Passport is locked to a single provider, we lose. The strategy is to be the standard user-facing layer across many providers — a stable surface for both AI products and end users. Open spec, open SDK, hosted control plane, and the user owns their Passport identity, not any one vendor.
Does my AI startup have to brand "Passport" in onboarding?
No. White-label mode lets you embed the consent and connection UI under your own brand. The user creates a Passport in your flow, not "via Passport." If they later use another product that supports Passport, that product can offer to import their existing Passport — but you're never required to surface our brand.
Where do raw credentials live?
With the provider, not with us. Composio, Arcade, Pipedream, Nango, and Zapier each maintain SOC 2-compliant credential vaults. Passport stores routing metadata, consent records, set definitions, and audit logs — never the tokens themselves. A Passport breach cannot expose user OAuth tokens.
Why not just use Composio directly?
Use Composio directly if you only need quick integrations and you don't care about portability or user-owned consent. Use Passport if you want users to bring approved access across multiple AI products, want to swap providers without rewriting agent code, or want a clean consent + audit surface that survives provider changes.
When does it ship?
Open beta in Q3 2026. The first slice is Composio-backed connections, the data model, the Connect UI, the MCP gateway, and audit logging. Additional adapters (Arcade, Pipedream, Nango, Zapier) follow based on what early adopters need first.
